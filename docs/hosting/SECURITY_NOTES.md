# Заметки по безопасности

Этот документ описывает безопасность текущего состояния сайта и требования к будущей админке.

## Текущее состояние

- Админки ещё нет.
- Логинов и паролей ещё нет.
- Записи в JSON ещё нет.
- `data/*.json` содержит публичный контент сайта: сотрудников, услуги и цены.
- `includes/` нужно закрывать от прямого доступа из браузера.

На Apache-compatible hosting прямой доступ к `includes/` должен закрываться файлом `includes/.htaccess`.

На Nginx `.htaccess` не применяется, поэтому запрет прямого доступа к `includes/` должен быть настроен на стороне хостинга.

## Что нельзя делать

- Не ставить права `777` без необходимости.
- Не давать публичный доступ к backups/logs.
- Не хранить пароль в открытом JSON.
- Не включать directory listing для служебных папок.
- Не открывать публичный доступ к будущим backup-файлам.

«Поставьте 777» - это не инструкция, а приглашение к ограблению.

## Будущая админка потребует

- Авторизацию.
- PHP sessions.
- CSRF-защиту.
- `password_hash` и `password_verify` для паролей.
- Права записи только в `data/*.json` и backup-папки.
- Backup перед каждым сохранением.
- Audit log для действий администратора.
- Запрет directory listing.
- Защиту `admin/`.
- Защиту `data/backups/`.
- Защиту `logs/`.

## Минимальная политика доступа для будущей админки

- Публичная часть сайта только читает данные.
- Админка записывает только разрешённые JSON-файлы.
- Backup создаётся перед изменением данных.
- Логи и backup не доступны из браузера.
- Пароль не хранится в открытом виде.
- Ошибки PHP не выводятся посетителям на странице.

## Что проверить на хостинге

- Прямой доступ к `/includes/bootstrap.php` запрещён.
- Прямой доступ к `/includes/json-storage.php` запрещён.
- PHP имеет доступ на чтение к `data/*.json`.
- Для будущей админки можно будет ограничить запись только нужными папками.
- Есть способ смотреть PHP error log без вывода ошибок посетителям.
