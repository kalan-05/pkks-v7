# Безопасный откат

Откат нужен, если после загрузки хостинг открыл не ту версию сайта, PHP-render не заработал или после сохранения в админке нужно вернуть прежние JSON-данные.

Важно: не удалять `index.html`. Он нужен как fallback.

## 1. Если `/` открывает не то

Если `/` открывает старый `index.html` вместо `index.php`:

1. Проверить настройку порядка index-файлов.
2. Поставить `index.php` выше `index.html`.
3. Если панель не даёт изменить порядок, обратиться в поддержку хостинга.
4. Временно открыть `/index.php` напрямую для проверки PHP-версии.

Если нужно временно вернуть статическую главную страницу:

- поставить `index.html` выше `index.php`;
- или попросить поддержку хостинга вернуть приоритет `index.html`;
- или открыть `/index.html` напрямую.

## 2. Если `index.php` даёт 500

Если `/index.php` возвращает `500 Internal Server Error`:

1. Проверить версию PHP.
2. Проверить, включено ли JSON-расширение PHP.
3. Проверить, загружены ли `data/*.json`.
4. Проверить права чтения для `data/*.json`.
5. Посмотреть PHP error log в панели хостинга.
6. Временно использовать `/index.html`, пока причина не устранена.

Если перед загрузкой был сделан backup, восстановить файлы из backup.

## 3. Если админка не открывается

Если `/admin/login.php` открывается, но войти нельзя:

1. Проверить, создан ли `config/admin-auth.php`.
2. Проверить, что `admin_login` совпадает с введённым логином.
3. Сгенерировать новый `password_hash` и заменить `admin_password_hash`.
4. Проверить, что PHP может писать в `storage/login-attempts.json`.
5. Проверить PHP error log хостинга.

Если закрытые страницы открываются без сессии или runtime-файлы скачиваются напрямую, это не rollback-случай. Нужно остановить передачу и настроить запрет доступа на хостинге.

## 4. Где лежат JSON backup

Админка создаёт backup перед сохранением данных:

```text
data/backups/team/team-*.json
data/backups/services/services-*.json
data/backups/prices/prices-*.json
```

Backup-файлы являются runtime-данными и не коммитятся в репозиторий. В репозитории остаются только placeholder-файлы для самих backup-папок.

## 5. Как восстановить сотрудников

1. Остановить редактирование в админке на время rollback.
2. Сделать копию текущего файла `data/team.json`.
3. Найти нужный backup в `data/backups/team/`.
4. Скопировать выбранный `team-*.json` поверх `data/team.json`.
5. Проверить главную страницу и `/admin/team.php`.
6. Убедиться, что кириллица не повреждена.

Rollback выполняется вручную на хостинге через файловый менеджер, SSH или другой инструмент, который предоставляет хостинг.

## 6. Как восстановить услуги

## 8. Если PHP не поддерживается

Если хостинг не поддерживает PHP:

- PHP-render не будет работать;
- JSON-данные не будут подставляться через `index.php`;
- админ-панель работать не сможет.

Временный вариант - открыть статическую fallback-версию через `/index.html`. Для полноценной работы нужен PHP-хостинг.

## 9. Если `.htaccess` ломает сайт

Если после загрузки `.htaccess` сайт перестал открываться:

1. Временно убрать `.htaccess` из корня сайта.
2. Проверить `/index.php` напрямую.
3. Уточнить у поддержки, поддерживается ли `.htaccess`.
4. Если `.htaccess` поддерживается частично, попросить поддержку настроить приоритет `index.php` перед `index.html`.
5. Отдельно проверить запрет доступа к `config/`, `storage/`, `data/backups/` и `includes/`.

На Nginx `.htaccess` не применяется, поэтому настройка index-файлов и запрет доступа к runtime-файлам должны выполняться на стороне хостинга.

## 10. Если нужно временно вернуть статический сайт

Безопасные варианты:

- открыть `/index.html` напрямую;
- временно поставить `index.html` выше `index.php`;
- временно убрать `.htaccess`, если он мешает работе хостинга;
- попросить поддержку хостинга вернуть приоритет `index.html`;
- восстановить файлы из backup.

После временного отката нужно сохранить `index.php`, `data/`, `includes/`, `admin/`, `config/` и `storage/`, чтобы PHP-версию и админку можно было вернуть без повторной сборки.
